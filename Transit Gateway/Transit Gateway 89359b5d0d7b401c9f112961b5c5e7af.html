1. Transit gateway 란?AWS Transit Gateway Web

Transit gateway의 자세한 설명은 AWS 공식 웹을 통해서 확인 부탁드립니다!

VPC 간 통신과 On-Premises, VPN을 Transit gateway에 연결하여 인프라 간소화, 중앙에서 관리할 수 있는 설루션입니다.VPC Peering은 VPC당 On-Premises를 연결해야 했지만, 한 개의 Transit gateway으로 On-Premises를 연결하여 각 VPC로 라우팅 할 수 있으며 중앙에서 Route 하여 관리 및 VPC Peering보다 단순한 인프라구조를 가져갈 수 있습니다. 하지만, Transit gateway는 VPC peering 보다 가격이 비싸며 단순간 구조에서는 VPC peering 좋지 않을까 합니다.Transit gateway 사용 시 VPC별로 Routing 설정하여 안전한 구조를 가져갈 수 있습니다. Transit gateway 예시 :  HQ User -> VPN Connection -> TGW -> EC2 Instance(Transit gateway Routing를 이용하여 HQ User는 VPN으로 EC2 Instance에 접근 가능하지만 EC2 Instance는 반대로 접근이 불가능)

3. Transit Gateway 생성

Create Transit Gateway 클릭해서 생성하자

Name Tag : TGW Name TagVPN ECMP support : VPN를 사용하는 경우 체크 Default route table association : VPC 또는 VPN에 association를 Default Route Table에 추가  Default route table propagation : VPC 또는 VPN에 propagation를 Default Route Table에 추가  Multicast support : 멀티캐스트 지원 여부Auto Accept shared attachments : 다른 계정과 공유하는 경우 체크

Pending 이후 Available로 변경된다.

Attachments 설정 VPC가 2개라면 2개를 생성한다.

설정했던 TGW ID를 클릭하자.

Attachment name tag 설정 후 맞는 VPC를 선택 하자

선택한 VPC AZ와 Subnet 설정하자 (Publie Subnet과 Private Subnet가 있다면 Private Subnet으로 해주자)

TGW 연결 시 ENI가 생성된다.

역시 이놈도 Pending 이후 Available로 변경된다.

Associations :  소속된 Attachment를 가리키며 각 Attachment는 Associations에 소속되어 있어야 Routes에 적용될 수 있다. Propagations : TGW의 Static Routing, Route Table를 전파한다. Routes : Propagation 된 Attachment들이 기본적으로 Apply 되며 다른 Route Table과 Static으로 Routes를 Propagation 할 수 있다.(생성한 Attachment가 Associations탭에 소속되어야만 Routes에 적용될 수 있고 Propagations에 소속된 Attachment들은 Route Table를 전파되어 Routes에 Default Applye 되며 Attachment가 되지 않는 녀석들을 Associations 녀석들과 Static으로 Routing 설정할 수 있다.)

여기까지 TGW의 Route 내용은 완료되었다..
